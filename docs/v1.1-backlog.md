## 1. Conversation branch-tree UI

**Status:** Backend complete; frontend tree view not built.

**Where:**
- Backend Tauri commands already exist:
  `src-tauri/src/commands/conversations.rs`:
  - `fork_message(conversation_id, parent_id, content) -> String`
  - `list_message_branches(conversation_id) -> Vec<MessageBranchNode>`
  (`MessageBranchNode { message_id, parent_id, role, created_at }`)
- `conversations.active_branch_message_id` column exists;
  `biscuitcode_db::Database::touch_conversation` already accepts an
  explicit leaf.
- No `chat_set_active_branch` Tauri command yet — callers would need to
  reach for `touch_conversation` via the existing handle, or a new thin
  command.

**Why deferred:** Phase 8 coder implemented the backend data layer but
no prior phase scaffolded a tree-view React component. Authoring a
tree-view UI unsupervised risks visual/UX regressions that only show
up on a real display; better held for a human-in-the-loop pass.

**Done looks like:**
- A new `src/components/BranchTreePanel.tsx` (or popover) that calls
  `list_message_branches`, renders the DAG, highlights the current
  `active_branch_message_id`, and on node click calls a new
  `chat_set_active_branch` command + refreshes `chat_list_messages`.
- An integration point in `ChatPanel.tsx` header (git-branch icon).
- Vitest coverage for the tree rendering + switch flow with mocked
  Tauri IPC.

## 3. `agentStore` dual-name API cleanup

**Status:** Tech debt. The store exports two parallel sets of action
names for the same operations:
- Primary (Phase 5/6a-i original): `startCard`, `appendArgsDelta`,
  `endCard(id, argsJson, result, error)`
- Plan-required aliases (6a-i): `addCard`, `updateCardArgs`,
  `completeCard`, `errorCard`

**Where:**
- `src/state/agentStore.ts` defines both.
- `src/components/ChatPanel.tsx` has **two** event-subscription paths:
  - `agent:event` listener (Phase 6a-ii) uses the new names.
  - `biscuitcode:chat-event:<convId>` listener (Phase 5) uses the
    old names.
- `tests/unit/agent-activity-panel.spec.tsx` uses the old names.
- `tests/unit/agent-event-bridge.spec.ts` and
  `tests/unit/agentStore.spec.ts` use the new names.

**Why deferred:** `endCard(id, argsJson, result, error)` takes a
boolean error flag and an argsJson string the new aliases don't expose.
Cleaning up requires either (a) enriching `completeCard` / `errorCard`
to take argsJson, or (b) converging both sets to a single set with
richer signatures and updating all call sites + tests. Either is
mechanical but needs careful semantic preservation.

**Done looks like:** One set of names, all call sites migrated, tests
updated, no behavior change. See Phase 6a-i Execution Notes for the
reviewer's context.

## 4. `workspace_trusted` server-side reading

**Status:** Hard-coded `false` in `src-tauri/src/commands/chat.rs` at
the `ExecutorContext` construction site (~line 346). The frontend
settings UI writes `workspace.trust` to localStorage; the Rust agent
path never reads it.

**Why deferred:** Reading localStorage from Rust needs a new Tauri
command (frontend calls it with the current workspace path, Rust
caches the result) or a `settings.json`-on-disk pattern. Not wired.

**Effect on v1.0:** Every write/shell tool call prompts confirmation,
even on workspaces the user marked trusted. Safer default, mildly
annoying for trusted workflows.

**Done looks like:**
- A `get_setting<T>(key)` Tauri command (or similar) backed by the
  settings store.
- `chat.rs` reads `workspace.trust.<pathHash>` and passes it through
  to `ExecutorContext::workspace_trusted`.
- Covered by an integration test showing that trusted workspaces skip
  the confirmation modal.
