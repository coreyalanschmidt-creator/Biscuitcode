BiscuitCode Icon Concepts — Design Reference

Three directions for the BiscuitCode application icon. Each is shown at multiple sizes on dark and light backgrounds to verify legibility down to 16×16 (system tray size). Brand colors are locked: biscuit-gold #E8B04C on cocoa-dark #1C1610.

Primary recommendation: Concept A ("The Prompt") — the >_ glyph reads instantly as a developer tool, scales cleanly to 16px, and sits comfortably in a launcher grid alongside VS Code, Warp, Zed, and Cursor without copying any of them.

Strong alternative: Concept C ("The Biscuit") — leans into the name with a literal biscuit-shape carrying the >_ glyph. More ownable and memorable, slightly more playful. Either is a defensible choice.

Design Notes

Glyph: chevron > + underscore _. Reads universally as "terminal prompt" / "developer tool."

Construction: chevron is a polyline with stroke-linecap:round; stroke weight scales up at smaller sizes (48px → 72px in viewBox) to maintain visual presence. Corner radius also scales down (22% → 18.75%) so the rounded square doesn't become a circle at 16px.

Why it works: professional silhouette, legible at every size, ownable, uses brand tokens verbatim. Safe pick — this is what launcher-grid peers expect.

Design Notes

Glyph: curly braces { } wrapping a solid biscuit-gold dot — "code block containing a value."

Trade-off: universally legible as "code" but less ownable — many projects use braces. The dot in the middle is the differentiating element.

16px caveat: braces get spindly below 24px. Stroke weight is bumped aggressively at the smallest sizes. Concept A is more robust at tray sizes.

Design Notes

Glyph: round biscuit shape with a radial-gradient "baked" face, darker outer edge, twelve perimeter pricks (digestive-biscuit style), and the >_ prompt in deep cocoa at the center.

Why it works: maximum ownability — no other dev tool looks like a biscuit. Memorable in a launcher grid (the round silhouette breaks up a row of rounded squares). Thematic without being cutesy.

Scaling strategy: pricks and baked edge stroke are decorative — rendered at 128px and above, dropped at 64px and below to avoid noise. At 16px, only the biscuit face + chunky >_ glyph remain.

Trade-off: slightly more "indie / playful" than Concept A. Excellent fit for BiscuitCode specifically, less of a generic professional statement.

How to use this file with Claude Code

  1. Pick a concept. Open this file in a browser to evaluate at real pixel sizes. The 16px preview is the make-or-break size — it must stay legible in the XFCE system tray.
  2. Attach to Claude Code. Drag this HTML into the chat when you start the build, alongside the super prompt. Say: "Use Concept A as the icon master" — or "Use Concept C" if you want the thematic biscuit. Render to SVG and all required PNG sizes for the .deb package.
  3. Let Claude Code extract the SVG. The hero SVGs inline in this file are production-ready — Claude Code can lift the chosen one directly into packaging/icons/biscuitcode.svg and rasterize the PNG set via rsvg-convert.
  4. Override anything. E.g., "Use Concept C but make the pricks slightly larger" or "Use Concept A but make the underscore shorter and closer to the chevron." Claude Code will adjust the SVG directly.
